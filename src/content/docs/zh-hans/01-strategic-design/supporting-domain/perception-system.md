## 1. 定位

感知系统是 Vox 的“感官”。它让共识体从只能响应显式指令的问答机器，进化为能理解用户状态、在合适时机以恰当方式主动提供支持的数字伙伴。

感知系统在架构中的位置：

- **数据采集**：由化身所在的端侧环境提供标准化、脱敏的传感器数据流
- **数据融合**：由 L0 层的感知模块将碎片化数据融合为高级上下文状态
- **决策触发**：由 Vox 基于上下文状态和原则中心格，判断是否介入、如何介入

感知系统本身不做决策。它只回答“用户现在处于什么状态”——是否介入、说什么、怎么说，是 Vox 的责任。

## 2. 感知原则

所有感知行为遵循四条最高准则。这些准则不是功能需求，而是伦理约束——它们直接贯彻了主权原则和可托付性原则。

### 2.1 感知不等于监视

系统只采集形成上下文所必需的最小数据量。不记录原始行为内容（如浏览的网页标题、键入的文本、正在观看的视频）。数据在端侧完成脱敏后才进入感知融合管线，原始传感器数据用完即弃，不落盘，不传输。

### 2.2 静默是美德

Vox 默认不主动发声。感知系统的存在不是为了“抓住每一个可以互动的机会”，而是为了在用户真正需要时，不错过那一个时刻。大多数时候，感知系统在安静地运行，用户感受不到它的存在。

### 2.3 赋能而非塑造

感知系统的目标是帮助用户更好地理解自己的状态、做出更高质量的决策——而非将用户的行为引导至某个预设的方向。这意味着：

- 系统可以提示“你已经在当前任务上停留了 2 小时，需要休息一下吗？”，但不能说“你应该休息”
- 系统可以呈现注意力热力图，帮助用户自我反思，但不能将图作为评判依据
- Vox 的任何感知触发的建议都标记为“建议”，用户可一键忽略

### 2.4 完全知情与控制

用户拥有所有感知维度的：

- **知情权**：每个感知维度的数据源、采集方式、用途，在系统中明文说明
- **开关权**：任一维度可被单独关闭，关闭后该维度不采集任何数据
- **验证权**：用户可随时查看当前感知状态（“系统现在认为我处于什么状态？”）
- **删除权**：感知融合生成的高级上下文状态可被删除，不进入长期记忆

## 3. 感知维度

系统融合四个维度的数据流，构建动态的“上下文意识”模型。每个维度的数据采集和脱敏在端侧完成。

### 3.1 时间感知

| 项目 | 说明 |
|:---|:---|
| **数据源** | 系统时钟、用户配置的日程（可选接入）、历史行为模式（本地分析，不外传） |
| **脱敏方式** | 不读取日程的具体内容（如“与某人开会”），仅提取时间段标签 |
| **产出** | 时段标签（“清晨 6-8”、“深夜 23-05”）、节律偏移标记（比平时晚睡 2 小时）、周期模式（每周日下午为“深度阅读时段”） |
| **用途** | Vox 根据时段调整交互风格（深夜轻声，白天正常）；识别节律偏移时，可选择温和提醒 |

### 3.2 注意力感知

| 项目 | 说明 |
|:---|:---|
| **数据源** | 活跃应用窗口的分类（不读标题）、键盘/鼠标输入事件的频率模式、当前化身的前台/后台状态 |
| **脱敏方式** | 不记录“正在浏览 X 网站”，仅记录活动类别（“信息检索”、“创作”、“通信”、“娱乐”、“系统操作”）；不记录键入内容，仅记录输入频率和节奏 |
| **产出** | 注意力状态标签（“专注”——持续输入中、“碎片”——频繁切换窗口、“卡住”——长时间无操作或同一段落反复修改、“流动”——稳定的浏览或阅读节奏） |
| **用途** | “卡住”状态可触发 Vox 温和询问；专注状态则保持静默 |

### 3.3 位置与环境感知

| 项目 | 说明 |
|:---|:---|
| **数据源** | 网络类型（WiFi/蜂窝/有线）、连接的设备类型、时区 |
| **脱敏方式** | 不使用 GPS 精确位置，不记录 WiFi SSID（仅用“已知网络 / 未知网络”标记），不记录 IP 地址 |
| **产出** | 环境标签（“家”、“办公室”、“移动中”、“公共场所”）、网络稳定性评估 |
| **用途** | 公共场所自动降低语音交互默认音量；移动中简化 Vox 输出为可快速浏览的短句；网络不稳定时优先本地处理 |

### 3.4 任务感知

| 项目 | 说明 |
|:---|:---|
| **数据源** | 第二大脑中当前 Pin 的项目、活跃文档的元数据（类型、标签、最后修改时间）、每日日志中的今日焦点 |
| **脱敏方式** | 不读取文档正文内容，仅访问结构化的元数据层 |
| **产出** | 当前任务上下文（“正在处理项目 A 的文档撰写阶段”）、任务切换频率、Pin 完成率趋势 |
| **用途** | 当用户在多个项目间频繁切换时，Vox 可建议重新评估当日 Pin 优先级；当 Pin 完成率持续偏低时，可在次日早晨温和询问是否需要调整计划 |

## 4. 感知融合

各维度的脱敏数据流由 L0 层的感知融合模块合并为**高级上下文状态**。融合不是简单的标签拼接，而是一个推理过程。

### 4.1 融合逻辑

融合模块将四个维度的产出作为输入，输出一个或多个**场景假设**及其置信度。当多个维度的数据指向同一场景时，置信度上升。

### 4.2 融合示例

| 时间 | 注意力 | 环境 | 任务 | 融合结果 | Vox 行为 |
|:---|:---|:---|:---|:---|:---|
| 深夜 | 专注 | 家 | 写作 | **可支配的创作时间** (置信度: 高) | 完全静默，不打扰 |
| 深夜 | 碎片 | 家 | 无明确任务 | **可能的睡前放松** (置信度: 中) | 静默，除非用户主动发起 |
| 午后 | 卡住 | 办公室 | 反复修改同一段落 | **可能的完美主义循环** (置信度: 中) | 可选询问：“需要另一个视角看看这段文字吗？” |
| 清晨 | 专注 | 家 | Pin 已完成 2/3 | **高效晨间工作** (置信度: 高) | 静默，完成后给出正向反馈 |
| 傍晚 | 碎片 | 移动中 | 频繁切换项目 | **可能的决策疲劳或注意力分散** (置信度: 低) | 不介入，记录模式供用户晚间复盘参考 |

### 4.3 误判防护

融合模块的每一个场景标签都附有置信度。只有置信度≥中等的场景，Vox 才会考虑介入。置信度为低的场景仅被记录为“模式数据点”，供用户主动查看复盘时参考，不触发任何主动交互。

用户可以在任意时刻查看“系统现在认为我处于什么状态”，并纠正误判。每一次纠正都反馈给融合模块，用于调整该用户的状态推理模型。

## 5. 与 Vox 和 Harness 的集成

### 5.1 感知→Vox

感知模块输出的上下文状态，作为 Vox 推理时的额外输入参数——与用户输入、对话历史、原则中心格一起，参与多元思维议会的推理过程。Vox 在生成建议时，议会中的情感派脑会特别关注上下文中的情绪和疲劳信号。

### 5.2 感知→Harness

感知输出也触发 Harness 的硬约束：

- **静默约束联动**：在感知判定为“可支配的创作时间”时，Harness 自动提升 Vox 主动介入的阈值——即使 L2 认为“用户可能需要帮助”，也必须保持静默
- **时间约束联动**：深夜时段，感知系统确认用户确实处于“睡前状态”而非“深夜加班”，则 Harness 硬约束禁止 Vox 发起非紧急交互
- **审计保护**：感知系统产生的上下文状态同样受 Harness 审计日志覆盖，用户可查证系统在何时基于什么上下文做出了什么判断

## 6. 用户控制面板

用户可在设置中查看和调整感知系统的运行状态：

| 控制项 | 选项 | 默认值 |
|:---|:---|:---|
| 感知系统总开关 | 开启 / 仅被动响应 / 关闭 | 开启 |
| 时间感知 | 开启 / 关闭 | 开启 |
| 注意力感知 | 开启 / 关闭 | 开启 |
| 位置与环境感知 | 开启 / 关闭 | 开启 |
| 任务感知 | 开启 / 关闭 | 开启 |
| Vox 主动介入频率 | 频繁 / 标准 / 仅重要 / 从不 | 标准 |
| 当前感知状态 | 实时显示系统对你的当前判断 | - |
| 感知历史 | 查看过去 7 天的上下文状态记录 | - |

所有控制项的修改即时生效，无需重启。
