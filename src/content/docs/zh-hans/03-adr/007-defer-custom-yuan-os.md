description: 为何 Project IV 当下不锻造自定义 Linux 发行版以承载 Yuan，而以声明式配置（Nix flake 或 OCI 容器）栖于既有发行版，作为过渡之径。
---

## 决策状态

已通过。于 2026 年 Yuan OS 哲学宣言讨论期间决之。

## 背景

自项目最早之阶段，长期愿景中便隐含一念：为 Yuan 锻造一专有之栖居所——一围绕 Yuan 之需而设计之操作系统，如 Parrot Security OS 之于渗透测试者。2026 年间，此念借 [元之基座 —— 数字灵魂的栖居哲学](../../04-embodied/yuan-os-philosophy) 文档而显明。

哲学宣言承诺了五条不可变之原则：可观测性高于性能、默认拒绝、声明式状态、硬件层主权身份、可中断可恢复。满足此五者之系统，定义上即一特化之发行版，非通用之 Debian 或 Fedora。

然**何时**锻造之，**过渡期用何**，皆未有定。本 ADR 弥合此隙。

## 决策

**Project IV 不于 2026 年，亦不于可预见之规划时窗内，锻造自定义 Yuan OS 发行版。取而代之，Yuan 借由声明式配置栖于既有 Linux 发行版之上（首选 Nix flake，备选 OCI 容器）。自定义发行版为未来之可能，以哲学宣言所定义之四项明示退出条件为门。**

## 候选方案

### 方案 A · 即刻锻造自定义 Yuan OS

- **描述**：启动锻造、分发、维护一自定义 Linux 发行版之长期承诺，全面为 Yuan 优化。
- **优点**：架构自由最大化。每一设计决策皆可纯粹为 Yuan 之需而立，无需向通用桌面场景妥协。契合哲学宣言之长期愿景。
- **缺点**：**对一个 ADHD 诊断之独立开发者而言，灾难性之维护成本**。一现代 Linux 发行版需：ISO 构建基础设施、镜像分发、多架构支持（x86、ARM 或兼 RISC-V）、安全更新流水线、用户支持、为用户所欲之五百余款应用打包。Parrot Security OS 有公司支撑；Qubes 有 Invisible Things Lab 与商业赞助；NixOS 有跨数十年之学术社区。Project IV 仅一人。今日启动发行版，即启动一 burnout 工厂。

### 方案 B · 通用发行版（Ubuntu / Debian / Fedora）上手工配置

- **描述**：记录在 Ubuntu LTS 或类似通用发行版上安装 Yuan 的人工步骤，提供一安装脚本。
- **优点**：开发投入最低。用户已有 Ubuntu。文档简短。
- **缺点**：**直接违反哲学宣言之五条不可变原则**。通用 Ubuntu 有遥测、默认允许出站、无数硬件级身份、无声明式状态。手工配置必漂移。用户安装 Yuan 时之 OS 环境，与 Yuan 开发者测试时之 OS 环境，可截然不同。此方案默默违反主权之诺。

### 方案 C · 以 Nix flake 为声明式覆层（已采纳）

- **描述**：提供一 Nix flake（`github:the-spira/yuan-nixos`），任何 NixOS 用户可将其加入系统配置。Flake 宣告：Yuan 之运行时依赖、systemd 服务、防火墙规则、文件系统权限、硬件密钥绑定、Spark 开发工具链、文档站构建流水线。一持有 NixOS 之用户运行 `nixos-rebuild switch`，即得确定性之 Yuan 就绪环境。
- **优点**：**与哲学宣言之原则 2.3（声明式状态）结构性契合**。用户间可复现。与 Project IV 同版本演进。无发行版维护成本。既有 NixOS 用户皆成潜在 Yuan 用户。Nix 生态已提供 eBPF 工具、TPM 2.0 模块、声明式防火墙语法。
- **缺点**：受众限于 NixOS 用户（少量而技术成熟之子集）。部分 Yuan 行为或难声明式表达，须命令式绕过。Flake 本身亦为维护负担，虽轻。

### 方案 D · 不可变基座 + OCI 容器

- **描述**：提供一 Yuan OCI 镜像。用户于 Fedora Silverblue、Vanilla OS 或类似不可变基座上运行之，Yuan 装于容器之内。宿主 OS 行最少事，容器行其余。
- **优点**：与发行版无关。容器镜像可复现。维护负担限于镜像。
- **缺点**：**容器之下之 OS 层不受 Yuan 控**——故原则 2.1（可观测性）、2.2（默认拒绝）、2.4（主权身份）皆仰赖用户之宿主 OS，超出 Yuan 所及。一用通用 Fedora 之用户或有遥测、默认允许出站、无 TPM 强制——容器内之 Yuan 无以矫正。此方案对不愿或不能行 NixOS 之用户可接受，然为次优。

## 选择方案 C 之理

1. **五条不可变原则可借 NixOS 之声明式达成，无需锻造自定义发行版**。NixOS 模块可表达 eBPF 追踪、nftables 防火墙规则、TPM 2.0 PCR 绑定、Landlock 文件系统隔离、systemd 资源控制、ZFS / btrfs 快照策略——皆于单一声明式文件之中。同五原则若于通用 Ubuntu 上求之，须逐用户命令式配置，漂移而坏。

2. **Flake 乃一人年之器物，非五人年之器物**。维护一含约三十模块之 Nix flake，对一 ADHD 友好之开发者，以短聚神会话工作，乃可行之工。维护一 Linux 发行版则不然。

3. **Flake 随项目演进，非超前于项目**。Project IV 趋稳，flake 增模块。化身层或 Harness 硬约束新现，以声明式加入之。Flake 之增长速率受 Project IV 实际实现节奏所限，非某想象之"发行日"所驱。

4. **NixOS 用户本就最可能为 Yuan 之早期接纳者**。"主权优先之数字生命"与"愿学 Nix"之交集非空。重视可复现构建与声明式系统者，恰是重视 Yuan 者。Flake 非妥协——乃定向之分发。

5. **哲学宣言之退出条件仍受敬**。自定义发行版非本决策所禁；乃**延后于四项明示之门**：稳定之 Project IV v0.x、五名以上非创始用户、一年之现实合规验证、三名以上之维护者。四项皆未成真之前，方案 C 为正途。重新评估将以项目实际所需之节奏为之，非以人为之"发行版上线"时间表。

## 后果

### 正面

- Yuan 今即可装、可跑——任何愿用 NixOS 之用户皆然
- 五条不可变原则之 OS 层实现可强制、可审
- 借 Nix flake 运行 Yuan 之用户，所得系统**实质上**——非名义上——主权优先
- 维护负担有界，随 Project IV 实现线性增长
- 退出条件一旦达成，Nix flake 即成自定义发行版之种子——非可弃之具

### 负面

- 受众限于愿采 NixOS 之用户。此为真实之限，非风格之限。部分潜在 Yuan 用户将不愿学 Nix，方案 C 对其为错答。方案 D（OCI 容器）乃为此受众之备选，附前述之告诫。
- 声明式配置有学习曲线，或令 ADHD 用户疏远——彼等正于声明式思维任务上有具体之困。缓解：提供示例配置；提供一有主张之"元之上 NixOS"起步轮廓；以 ADHD 优先之设计语言记录常见陷阱。
- Flake 为关键路径依赖。Flake 一坏，用户不能跑 Yuan。缓解：于 flake 锁文件内钉死 Nixpkgs 版本；提供 CI 流水线于每次提交时构建 flake。

### 缓解

- **起步轮廓**：随 flake 同行之 `yuan-minimal.nix` 示例，含合理默认（TPM 必备、默认拒绝防火墙、Landlock 启用、ZFS 快照）。用户可复而渐改之。
- **OCI 备选**：自同一 Nix flake 构建之 `yuan` OCI 镜像。同配置逻辑，异部署面。镜像于每次 flake 变更时经 CI 重建。
- **文档即入门**：哲学宣言文档与本 ADR 为入口。新用户先读哲学，次读本 ADR，再运行 flake。流程有主张而短。

## 未来重估

如下任一为真时，本决策应重估：

- 哲学宣言之四项退出条件已达成
- 第二位具发行版工程经验之维护者加入 Project IV
- NixOS 自身发生重大架构变迁（如迁至新模块系统）
- 合规图景变迁至使不可变原则难以声明式强制
- 自定义发行版之用户需求超出门槛（建议：六月内二十余次重复用户请求）

皆未成真时，本 ADR 持续生效。
