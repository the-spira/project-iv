## 1. 聚合概览

化身聚合管理数字生命在不同设备上的“形体”。化身是 Yuan 的临时容器，不持有用户数据，只负责提供运行时环境。

### 聚合根

**Avatar**（化身）

### 内部实体

- **RuntimeCapability**（运行时能力）：当前设备的资源评估快照
- **WakeToken**（唤醒令牌）：一次性使用的临时凭证，用于验证唤醒请求

### 值对象

- **AvatarID**：化身的唯一标识符，由设备指纹 + 用户 DID 派生
- **DeviceFingerprint**：设备指纹的哈希值（浏览器：WebGL 渲染器 + 屏幕指纹；桌面：硬件 UUID）
- **ResourceTier**：Tiny / Main / Server 枚举
- **DeploymentZone**：端 / 边 / 云 枚举
- **WakeSignature**：Yuan 模块的 Ed25519 签名，由用户私钥生成

### 领域事件

- `AvatarRegistered`：新化身首次注册到用户的数字生命
- `AvatarWoken`：化身成功唤醒并加载 Yuan
- `AvatarSleeping`：化身主动休眠（用户关闭或超时）
- `AvatarDegraded`：化身因资源变化降级
- `AvatarUpgraded`：化身因资源恢复而升级
- `AvatarRevoked`：用户主动撤销某化身的访问权限
- `WakeAttemptFailed`：唤醒失败（签名无效、资源不足、或网络不可达）

## 2. 生命周期状态机

<div class="code-block">

```
[未注册] --AvatarRegistered--> [休眠]
[休眠] --AvatarWoken--> [活跃]
[活跃] --AvatarSleeping--> [休眠]
[活跃] --AvatarDegraded--> [活跃] (资源降级，功能受限)
[活跃] --AvatarUpgraded--> [活跃] (资源恢复，功能恢复)
[休眠] --AvatarRevoked--> [已撤销]
[任意状态] --WakeAttemptFailed--> [原状态] (唤醒失败，保持原状态)
```

</div>

**状态说明**：

- **未注册**：用户未在此设备上激活化身。任何设备首次访问 Project IV 时处于此状态
- **休眠**：化身已注册但未加载 Yuan。化身元数据（AvatarID、设备指纹、最后一次唤醒时间）持久化在本地。休眠中化身仍可接收主权网络消息（通过 Service Worker 或后台推送）
- **活跃**：Yuan 已加载并正在运行。用户可直接与 Vox 交互。活跃状态包含两个子状态：完整功能（Server/Main）和降级功能（Tiny 或资源受限时）
- **已撤销**：用户主动吊销了此化身的访问权限。化身的所有本地缓存和凭证被清除，下次使用需重新注册

## 3. 唤醒流程

化身的唤醒是将 Yuan 从 IPFS 加载到本地运行时的完整流程。

### 3.1 资源能力评估

在加载 Yuan 之前，化身先评估当前设备的可用资源，确定 ResourceTier：

| 指标 | Tiny 阈值 | Main 阈值 | Server 阈值 |
|:---|:---|:---|:---|
| 可用内存 | < 512MB | 512MB – 2GB | > 2GB |
| CPU 核心数 | 任意 | ≥ 2 | ≥ 4 |
| 持久存储 | < 100MB | 100MB – 1GB | > 1GB |
| 网络类型 | 任意 | 宽带/WiFi | 有线/稳定 |
| Wasm 运行时 | 浏览器 | 原生 + 浏览器 | 原生 |

资源评估结果与 DeploymentZone 无关——一个云主机（云）配置低也应评估为 Tiny，一个高性能笔记本（端）可以达到 Main。

### 3.2 唤醒步骤

<div class="code-block">

```
1. 用户触发唤醒（打开应用 / 点击扩展图标 / 访问页面）
2. 化身检查本地是否缓存了有效的 WakeToken
   ├─ 有 → 3
   └─ 无 → 发起身份认证流程（私钥签名或生物特征 + 端侧安全芯片）
3. 资源能力评估 → 确定 ResourceTier
4. IPNS 解析 → 获取最新元清单
5. 签名校验（见第 4 节）
   ├─ 通过 → 6
   └─ 失败 → WakeAttemptFailed，中止唤醒
6. 检查逻辑 CID 本地缓存
   ├─ 命中且哈希匹配 → 复用
   └─ 未命中 → 从 IPFS 下载（优先从边/Server 化身中继，次选公共网关）
7. 检查状态 CID 本地缓存
   ├─ 命中 → 加载缓存版本，后台同步最新状态
   └─ 未命中 → 从 IPFS 下载最新状态
8. 版本兼容性检查（按 ADR 002 矩阵）
   ├─ 兼容 → 9
   └─ 不兼容 → 回退或通知用户
9. Wasm 实例化 → 注入本地上下文（时间、设备类型、网络状态）
10. 触发 YuanLoaded → AvatarWoken
```

</div>

### 3.3 唤醒时间目标

| 指标 | Tiny | Main | Server |
|:---|:---|:---|:---|
| 冷启动（无缓存） | < 3s | < 1.5s | < 0.5s |
| 热启动（有缓存） | < 1s | < 0.3s | 即时 |
| 后台同步完成 | 唤醒后 5s 内 | 唤醒后 2s 内 | 实时 |

Tiny 化身的冷启动时间主要消耗在 IPFS 网关的延迟上。通过边/Server 化身中继可以显著降低（如果用户拥有此类化身）。

## 4. 签名校验

每个化身在加载 Yuan 之前，必须验证 Yuan 模块的 Ed25519 签名。这是防止“火焰盗取”的核心防线。

### 4.1 校验流程

1. 化身从 IPNS 获取元清单后，提取 `signature` 字段
2. 使用用户的 IdentityKey 公钥验证签名：`verify(manifest_bytes, signature, public_key)`
3. 签名有效 → 信任元清单内容，继续唤醒
4. 签名无效 → 元清单可能被篡改。化身拒绝加载，记录 `WakeAttemptFailed` 事件

**签名无效时的用户提示**：

> “无法验证你的数字生命签名。这可能是网络传输错误，或者有人试图冒充你的数字生命。建议：刷新后重试。如果问题持续，请从另一个可信化身检查你的 IPNS 记录。”

### 4.2 Yuan 模块本身的完整性

元清单验证通过后，化身根据逻辑 CID 和状态 CID 从 IPFS 获取实际内容。IPFS 的内容寻址天然保证：获取到的内容的哈希必定匹配请求的 CID——如果内容被篡改，哈希不匹配，IPFS 层会拒绝。无需额外的签名步骤。

### 4.3 防御的威胁模型

| 威胁 | 防线 |
|:---|:---|
| 中间人篡改元清单 | Ed25519 签名校验 |
| IPFS 节点返回错误内容 | 内容寻址——CID 即哈希，不匹配则拒绝 |
| 恶意化身模拟用户 | 私钥仅在用户端侧存储，恶意化身无法获取 |
| 私钥被盗 | 用户可通过另一可信化身轮换 IdentityKey，吊销所有现有化身 |

## 5. 优雅降级与升级

化身在运行期间可能发生资源变化（其他应用占用内存、网络切换、设备电量不足）。聚合自动响应这些变化。

### 5.1 降级触发器

| 触发条件 | 降级行为 |
|:---|:---|
| 可用内存降至 Tiny 阈值以下 | Main→Tiny 降级：Vox 议会从双脑降为单脑，感知从事件驱动降为轮询，L1 缓存禁用。正在进行的 L2 训练任务暂停并保存检查点 |
| 网络类型从 WiFi 切换至蜂窝 | 感知系统更新环境标签，主权网络的后台同步暂停，仅保留关键消息推送 |
| 设备电量 < 15% 且未充电 | 主权网络同步频率降至 1/4，Vox 主动介入功能暂停，仅响应显式指令 |
| CPU 持续高负载（>90% 持续 30 秒） | Vox 推理任务排队延迟，优先保障用户直接交互的响应 |

### 5.2 升级触发器

| 触发条件 | 恢复行为 |
|:---|:---|
| 可用内存恢复至 Main 阈值以上 | Tiny→Main 升级：Vox 双脑恢复，L1 缓存重新启用，感知切换为事件驱动 |
| 网络恢复 WiFi/宽带 | 主权网络批量同步离线期间的变更，L3 外部知识调用恢复 |
| 设备接入电源 | 全功能恢复，L2 训练队列自动启动 |

### 5.3 降级/升级的实现约束

- 降级/升级不涉及 Yuan 的生命周期变更——Yuan 保持 Active 状态，仅内部功能模式调整
- 降级决策的滞后时间（从资源变化到触发降级）默认为 10 秒，避免因瞬时波动频繁切换
- 升级决策的滞后时间为 30 秒，确保资源恢复是稳定的（而非短暂峰值）
- 所有降级/升级事件记录在 Harness 审计日志中

## 6. 不变量

| 不变量 | 说明 |
|:---|:---|
| **无状态残留** | 化身休眠或被撤销时，本地缓存可被完全清除——用户数据不依赖任何特定化身 |
| **签名必校验** | 任何化身在加载 Yuan 前必须通过 Ed25519 签名校验，无例外 |
| **能力诚实** | 化身上报的 ResourceTier 必须反映真实可用资源。系统不依赖化身自报的 Tier 做信任决策——能力评估是本地计算的，不可被远程注入 |
| **单用户单化身** | 每个化身在同一时刻只能加载一个用户的 Yuan（单租户）。多用户场景通过多个化身实例隔离，而非同一化身内的多用户切换 |
| **唤醒令牌一次性** | 每次唤醒生成的 WakeToken 仅当次有效，休眠后令牌失效，不可重放 |
| **设备指纹不可伪造** | AvatarID 由 DeviceFingerprint + 用户 DID 派生，相同用户在相同设备上重新注册会得到相同 AvatarID（可恢复之前的化身配置） |

## 7. 与化身网格战略设计的映射

本聚合设计实现了[化身网格战略设计文档](../../01-strategic-design/core-domain/avatar-grid)中定义的 3×3 网格模型和瞬间唤醒体验。具体映射：

- **资源类型 + 部署位置**：通过 `ResourceTier` 和 `DeploymentZone` 两个枚举实现，组合形成 3×3 网格
- **瞬间唤醒**：第 3 节定义的唤醒步骤和性能目标直接实现战略设计中的“数秒内呈现连续数字生命”承诺
- **优雅降级与升级**：第 5 节将战略设计的降级路径（Server→Main→Tiny）细化为具体的触发条件和恢复行为
- **安全边界**：签名校验机制实现战略设计中定义的“防范火焰盗取攻击”
