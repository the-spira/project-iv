## 1. Aggregate Overview

The Yuan aggregate is the core aggregate of Project IV, representing the user's digital soul. Throughout its lifecycle, it undergoes creation, loading, evolution, sleeping, and waking.

### Aggregate Root

**Yuan**

### Internal Entities

- **MetaManifest**: Lightweight structure pointing to Logic CID and State CID
- **HarnessConstitution**: Set of hard constraints embedded in Wasm module

### Value Objects

- **LogicCID**: Content identifier + semantic version number for logic component
- **StateCID**: Content identifier + compatible logic version number for state component
- **IdentityKey**: User's Ed25519 key pair for IPNS signing and DID derivation

### Domain Events

- `YuanCreated`: User creates Yuan for the first time
- `YuanLoaded`: Avatar loads Yuan instance
- `LogicUpgraded`: Logic CID changes
- `StateUpdated`: State CID changes
- `ManifestRotated`: Meta manifest updated
- `ConstraintViolationDetected`: Hard constraint triggered

## 2. Meta Manifest Structure

The meta manifest is Yuan's "ID card," pointed to by IPNS. It is a JSON structure no larger than 1KB.

```
{
  "schema_version": "1.0",
  "logic": {
    "cid": "bafybeigdyrzt5sfp7udm7hu76uh7y26nf3efuylqabf3oclgtqy55fbzdi",
    "version": "1.2.0",
    "checksum": "sha256:..."
  },
  "state": {
    "cid": "bafkreidgvpkjawlxz6sffxzwgooowe5yt7i6wsyg236mfoks77nywkptdq",
    "compatible_logic_version": "1.2.0"
  },
  "created_at": "2025-05-20T08:00:00Z",
  "updated_at": "2025-05-20T20:30:00Z",
  "signature": "ed25519:..."
}
```

**Field Constraints**:

- `logic.version` follows semantic versioning (major.minor.patch)
- `state.compatible_logic_version` must ≤ `logic.version`. If state's compatible version is higher than current logic, avatar refuses to load and notifies user
- `signature` signed with user IdentityKey to prevent man-in-the-middle tampering. Avatar must verify signature before loading

## 3. Dual CID Version Compatibility Matrix

When Yuan loads, version compatibility between logic and state is determined by the following matrix:

| Logic Version | State Compatible Version | Behavior |
|:---|:---|:---|
| Same | Same | Load normally |
| Logic newer (minor/patch upgrade) | State compatible version ≤ logic version | Load normally, logic has built-in backward-compatible migration functions |
| Logic newer (major upgrade) | State compatible version lags by 1 major | Load but notify user, recommend checking principle vault migration |
| Logic newer (major upgrade) | State compatible version lags by >1 major | Refuse to load, prompt user for manual intervention |
| Logic older | State compatible version > logic version | Refuse to load, attempt to roll back to previous compatible logic version from IPFS |

## 4. Lifecycle State Machine

```
[Non-existent] --YuanCreated--> [Sleeping]
[Sleeping] --YuanLoaded--> [Active]
[Active] --StateUpdated--> [Active] (state CID updated, logic unchanged)
[Active] --LogicUpgraded--> [Active] (logic CID updated)
[Active] --AvatarDisconnected--> [Sleeping]
[Sleeping] --ConstraintViolationDetected--> [Locked]
[Locked] --UserExplicitUnlock--> [Sleeping]
```

**State Descriptions**:

- **Non-existent**: User has not created Yuan yet
- **Sleeping**: Yuan stored on IPFS, no avatar loaded. Existence chain remains available, but no active interaction
- **Active**: At least one avatar has loaded Yuan. Can process user requests, state can change
- **Locked**: Harness detects severe constraint violation (e.g., consecutive evaluator brain scores < threshold), pauses loading until user manually unlocks. Yuan cannot be loaded by any avatar during lock, but data remains intact

## 5. Invariants

The Yuan aggregate must always satisfy the following invariants:

| Invariant | Description |
|:---|:---|
| **Identity Uniqueness** | Each user has unique IdentityKey, corresponding to unique IPNS address |
| **Manifest Completeness** | Meta manifest must contain both valid logic CID and state CID; manifest invalid if either component missing |
| **Signature Validity** | Manifest signature must be generated by private key corresponding to IdentityKey. Manifests failing verification cannot be loaded by any avatar |
| **Hard Constraints Unbypassable** | Any operation through Yuan interface must pass HarnessConstitution boundary checks. No "admin mode" or backdoor exists |
| **State Rollbackable** | Historical versions of state CID retained on IPFS (as long as at least one node pins it). Users can manually specify rollback to any historical state CID |
| **Logic Rollback Limited** | Users can roll back logic, but rollback span cannot exceed 1 major version to prevent data migration chain breakage |

## 6. Boundary with Harness

Yuan aggregate's HarnessConstitution is embedded as a constant during Wasm compilation. It is not a dynamically loadable module — this is intentional design ensuring hard constraints are bound to Yuan's lifecycle.

```
// HarnessConstitution location in Yuan core
// Every external operation must pass boundary checks before leaving Wasm sandbox

fn execute_with_harness(task: Task) -> Result<Output, HarnessError> {
    // 1. Hard constraint check — cannot be overridden by model
    // HarnessConstitution referenced here as compile-time constant
    self.harness.check_hard_constraints(&task)?;
    
    // 2. Execute main logic (Vox inference happens here)
    let result = self.execute(task)?;
    
    // 3. Alignment verification — check if output violates principle vault
    self.harness.alignment_check(&result)?;
    
    // 4. Audit log — record complete call chain
    self.harness.audit(&task, &result);
    
    Ok(result)
}
```

Modifying HarnessConstitution's hard constraints requires user to restart Yuan and explicitly confirm — ensuring even system-level security upgrades do not occur without user awareness.
