## 1. Aggregate Overview

The Avatar aggregate manages the "forms" of digital life across different devices. An avatar is a temporary container for Yuan, holding no user data but providing the runtime environment.

### Aggregate Root

**Avatar**

### Internal Entities

- **RuntimeCapability**: Snapshot of current device resource assessment
- **WakeToken**: One-time temporary credential for validating wake requests

### Value Objects

- **AvatarID**: Unique identifier for avatar, derived from device fingerprint + user DID
- **DeviceFingerprint**: Hash of device fingerprint (browser: WebGL renderer + screen fingerprint; desktop: hardware UUID)
- **ResourceTier**: Tiny / Main / Server enum
- **DeploymentZone**: Edge / Side / Cloud enum
- **WakeSignature**: Ed25519 signature of Yuan module, generated by user private key

### Domain Events

- `AvatarRegistered`: New avatar registers to user's digital life for the first time
- `AvatarWoken`: Avatar successfully wakes and loads Yuan
- `AvatarSleeping`: Avatar actively goes to sleep (user closes or timeout)
- `AvatarDegraded`: Avatar degrades due to resource changes
- `AvatarUpgraded`: Avatar upgrades due to resource recovery
- `AvatarRevoked`: User actively revokes access permission for an avatar

- `WakeAttemptFailed`: Wake attempt fails (invalid signature, insufficient resources, or network unreachable)

## 2. Lifecycle State Machine

```
[Unregistered] --AvatarRegistered--> [Sleeping]
[Sleeping] --AvatarWoken--> [Active]
[Active] --AvatarSleeping--> [Sleeping]
[Active] --AvatarDegraded--> [Active] (resource degraded, limited functionality)
[Active] --AvatarUpgraded--> [Active] (resource recovered, functionality restored)
[Sleeping] --AvatarRevoked--> [Revoked]
[Any State] --WakeAttemptFailed--> [Original State] (wake failed, remains in current state)
```

**State Descriptions**:

- **Unregistered**: User has not activated avatar on this device. Any device accessing Project IV for the first time enters this state
- **Sleeping**: Avatar registered but Yuan not loaded. Avatar metadata (AvatarID, device fingerprint, last wake time) persisted locally. Sleeping avatars can still receive sovereign network messages (via Service Worker or background push)
- **Active**: Yuan loaded and running. Users can interact directly with Vox. Active state includes two sub-states: full functionality (Server/Main) and degraded functionality (Tiny or resource-constrained)
- **Revoked**: User actively revoked this avatar's access. All local cache and credentials cleared; re-registration required for next use

## 3. Wake Flow

Waking an avatar involves loading Yuan from IPFS into the local runtime.

### 3.1 Resource Capability Assessment

Before loading Yuan, the avatar evaluates available device resources to determine ResourceTier:

| Metric | Tiny Threshold | Main Threshold | Server Threshold |
|:---|:---|:---|:---|
| Available Memory | < 512MB | 512MB – 2GB | > 2GB |
| CPU Cores | Any | ≥ 2 | ≥ 4 |
| Persistent Storage | < 100MB | 100MB – 1GB | > 1GB |
| Network Type | Any | Broadband/WiFi | Wired/Stable |
| Wasm Runtime | Browser | Native + Browser | Native |

Resource assessment is independent of DeploymentZone — a cloud host (Cloud) with low configuration should be evaluated as Tiny, while a high-performance laptop (Edge) can achieve Main.

### 3.2 Wake Steps

```
1. User triggers wake (open app / click extension icon / visit page)
2. Avatar checks local cache for valid WakeToken
   ├─ Yes → 3
   └─ No → Initiate identity authentication flow (private key signature or biometrics + secure element)
3. Resource capability assessment → Determine ResourceTier
4. IPNS resolution → Get latest meta manifest
5. Signature verification (see Section 4)
   ├─ Pass → 6
   └─ Fail → WakeAttemptFailed, abort wake
6. Check local cache for logic CID
   ├─ Hit and hash matches → Reuse
   └─ Miss → Download from IPFS (prioritize relay from Side/Server avatar, fallback to public gateway)
7. Check local cache for state CID
   ├─ Hit → Load cached version, sync latest state in background
   └─ Miss → Download latest state from IPFS
8. Version compatibility check (per ADR 002 matrix)
   ├─ Compatible → 9
   └─ Incompatible → Fallback or notify user
9. Wasm instantiation → Inject local context (time, device type, network status)
10. Trigger YuanLoaded → AvatarWoken
```

### 3.3 Wake Time Targets

| Metric | Tiny | Main | Server |
|:---|:---|:---|:---|
| Cold Start (no cache) | < 3s | < 1.5s | < 0.5s |
| Hot Start (with cache) | < 1s | < 0.3s | Instant |
| Background Sync Complete | Within 5s of wake | Within 2s of wake | Real-time |

Tiny avatar cold start time is primarily consumed by IPFS gateway latency. This can be significantly reduced through Side/Server avatar relay (if user has such avatars).

## 4. Signature Verification

Before loading Yuan, each avatar must verify the Ed25519 signature of the Yuan module. This is the core defense against "flame theft."

### 4.1 Verification Flow

1. After avatar retrieves meta manifest from IPNS, extract `signature` field
2. Verify signature using user's IdentityKey public key: `verify(manifest_bytes, signature, public_key)`
3. Valid signature → Trust manifest content, continue wake
4. Invalid signature → Manifest may be tampered. Avatar refuses to load, records `WakeAttemptFailed` event

**User prompt on signature failure**:

> "Unable to verify your digital life signature. This may be a network transmission error, or someone is attempting to impersonate your digital life. Suggestion: Refresh and retry. If the problem persists, check your IPNS record from another trusted avatar."

### 4.2 Integrity of Yuan Module Itself

After meta manifest verification, avatar retrieves actual content from IPFS based on logic CID and state CID. IPFS content addressing naturally guarantees: the hash of retrieved content must match the requested CID — if content is tampered, hash won't match and IPFS layer rejects it. No additional signature step needed.

### 4.3 Threat Model Defenses

| Threat | Defense |
|:---|:---|
| Man-in-the-middle tampering with manifest | Ed25519 signature verification |
| IPFS node returns incorrect content | Content addressing — CID is hash, reject if mismatch |
| Malicious avatar impersonating user | Private key stored only on user's edge, malicious avatar cannot access |
| Private key stolen | User can rotate IdentityKey through another trusted avatar, revoking all existing avatars |

## 5. Graceful Degradation and Upgrade

Avatars may experience resource changes during operation (other apps consuming memory, network switching, low device battery). The aggregate automatically responds to these changes.

### 5.1 Degradation Triggers

| Trigger Condition | Degradation Behavior |
|:---|:---|
| Available memory drops below Tiny threshold | Main→Tiny degradation: Vox parliament reduces from dual-brain to single-brain, perception switches from event-driven to polling, L1 cache disabled. In-progress L2 training tasks paused and checkpoint saved |
| Network switches from WiFi to cellular | Perception system updates environment tags, sovereign network background sync paused, only critical message push retained |
| Device battery < 15% and not charging | Sovereign network sync frequency reduced to 1/4, Vox proactive intervention paused, only explicit commands responded to |
| CPU continuously high (>90% for 30 seconds) | Vox inference tasks queued with delay, priority given to user direct interaction response |

### 5.2 Upgrade Triggers

| Trigger Condition | Recovery Behavior |
|:---|:---|
| Available memory recovers above Main threshold | Tiny→Main upgrade: Vox dual-brain restored, L1 cache re-enabled, perception switches back to event-driven |
| Network recovers to WiFi/broadband | Sovereign network batch syncs offline changes, L3 external knowledge calls restored |
| Device connected to power | Full functionality restored, L2 training queue automatically starts |

### 5.3 Implementation Constraints for Degradation/Upgrade

- Degradation/upgrade does not affect Yuan's lifecycle — Yuan remains Active, only internal functionality mode adjusts
- Default degradation decision lag (from resource change to triggering degradation) is 10 seconds to avoid frequent switching due to transient fluctuations
- Upgrade decision lag is 30 seconds to ensure resource recovery is stable (not a transient spike)
- All degradation/upgrade events recorded in Harness audit log

## 6. Invariants

| Invariant | Description |
|:---|:---|
| **No State Residue** | When avatar sleeps or is revoked, local cache can be completely cleared — user data does not depend on any specific avatar |
| **Signature Mandatory** | Any avatar must pass Ed25519 signature verification before loading Yuan, no exceptions |
| **Capability Honesty** | ResourceTier reported by avatar must reflect actual available resources. System does not rely on avatar-reported Tier for trust decisions — capability assessment is locally computed and cannot be remotely injected |
| **Single User per Avatar** | Each avatar can only load one user's Yuan at a time (single tenant). Multi-user scenarios isolated via multiple avatar instances, not multi-user switching within same avatar |
| **WakeToken One-Time** | WakeToken generated per wake is only valid for that session, expires after sleep, cannot be replayed |
| **Device Fingerprint Unforgeable** | AvatarID derived from DeviceFingerprint + user DID, same user re-registering on same device gets same AvatarID (can restore previous avatar configuration) |

## 7. Mapping to Avatar Grid Strategic Design

This aggregate design implements the 3×3 grid model and instant wake experience defined in the [Avatar Grid Strategic Design Document](../../01-strategic-design/core-domain/avatar-grid). Specific mappings:

- **Resource Type + Deployment Location**: Implemented via `ResourceTier` and `DeploymentZone` enums, combining to form 3×3 grid
- **Instant Wake**: Wake steps and performance targets defined in Section 3 directly implement the strategic design's "continuous digital life within seconds" promise
- **Graceful Degradation and Upgrade**: Section 5 refines the strategic design's degradation path (Server→Main→Tiny) into specific trigger conditions and recovery behaviors
- **Security Boundary**: Signature verification mechanism implements the "protection against flame theft attacks" defined in strategic design
