## 1. Positioning

Domain events are the core asynchronous communication contracts between bounded contexts in Project IV. Each event represents a meaningful moment in the existence of a digital life.

This document is the authoritative source for event definitions. Events serve as the raw data source for existence chains and audit logs.

## 5. Event Usage Conventions

### 5.1 Event Naming

All events use `PastTenseVerb` + `Noun` format, starting with a past tense verb (e.g., `YuanCreated`, `KnowledgeBlockUpdated`). Event names are self-explanatory — what happened can be understood solely from the name.

### 5.2 Minimal Data Carried by Events

Each event carries the minimum data sufficient to reconstruct context. Events are not logs — they carry key information about "what happened," while detailed context about "why it happened" is supplemented by audit logs. Specific rules:

- **Must include**: Unique identifier of the event subject, timestamp of event occurrence, causal relationship triggering the event (e.g., `cause`, `trigger_reason` fields)
- **Must not include**: Full content payloads (referenced via CID), personal identity information, directly identifiable user data

### 5.3 Event Immutability

Once generated, events cannot be modified. Event IDs are derived from their content hash. If the same business fact needs correction (e.g., user corrects perception system's misjudgment), generate a new event (`UserCorrectedContext`) rather than modifying the old one.

### 5.4 Event Subscription

Bounded contexts subscribe to event types they care about through the sovereign network's event bus:

| Subscriber | Subscribed Events |
|:---|:---|
| Harness Layer | All `*Constraint*`, `Alignment*`, `Reflection*` events |
| Audit Log | All events (full record) |
| Perception System | `ContextShiftDetected`, `UserCorrectedContext` |
| Entropy Management Agent | `KnowledgeBlock*`, `Principle*` |
| Vox | `ContextStateFused`, `ConflictDetected`, `UserVerdictGiven` |
| DaoKit Manager | `DaoKit*` |

### 5.5 Relationship Between Events and Existence Chain

The `*Created`, `*Updated`, and `*Extended` events of Yuan and Sovereign Network in the core domain serve as data sources for the existence chain. The existence chain itself is not an event, but a **materialized view** of these events — a singly linked list connected by `state_cid` change events.
